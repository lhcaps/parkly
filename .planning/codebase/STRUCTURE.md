## Naming Conventions

| Pattern | Example |
|---------|---------|
| API route modules | `modules/*/interfaces/http/register-*-routes.ts` |
| Service files | `modules/*/application/*-service.ts` |
| Feature folders | `features/*/` (kebab-case: `run-lane`, `review-queue`) |
| Page components | `*Page.tsx` (PascalCase) |
| Route paths | `/kebab-case` |
| Test files | `pr*.test.ts` (PR-level), `smoke/*.test.ts` (release gate) |
| Zod schemas | `*.schemas.ts` suffix |
| i18n keys | `route.*`, `label.*`, `error.*` dot notation |
| Role constants | `UPPERCASE_SNAKE_CASE` (e.g., `SUPER_ADMIN`) |

Structure is clean and follows monorepo conventions. API is well-modularized by domain. Web follows feature-based organization. Key concern: legacy page files (GatePage, DevicesPage, DashboardPage, GateEventsMonitorPage) exist as redirect targets but may be dead code.
